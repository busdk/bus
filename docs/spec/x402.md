# x402 (Hosted HTTP Feature)

## What this is
x402 is one HTTP capture mechanism for recording Bus transactions. It defines:
- a 402 response body (“payment required”) with accept options
- request/response headers carrying payment proof artifacts
- normalization into the micropayments `Transaction` model

Layering rule (binding):
- x402 is a **hosted HTTP feature** implemented in the hosted facilitator service (see `docs/spec/hosted-facilitator-http.md`).
- **Bus Core** provides generic primitives only: schema/unit IO, validation hooks, and state-backend operations.
- The hosted HTTP service owns x402-specific request parsing, 402 body generation, and ingest mapping, calling core primitives to persist records.

## Out of scope (v1 intent preserved)
- blockchain/network settlement
- on-chain confirmation

## Wire formats (binding; versioned JSON)

### 402 response body
JSON object:
- `x402Version` (number, default `1`)
- `accepts` (array)

Each accept option:
- `scheme` (string)
- `network` (string)
- `maxAmountRequired` (string)
- `resource` (string)
- `payTo` (string)
- `asset` (string)
- optional `description` (string)
- optional `maxTimeoutSeconds` (number)

### `X-PAYMENT`
Header contains base64-encoded JSON:
- `x402Version`
- `scheme`
- `network`
- `payload` (scheme-specific object)

### `X-PAYMENT-RESPONSE`
Header contains base64-encoded JSON:
- `success` (boolean)
- `transaction` (string; external ref)
- `network` (string)
- `payer` (string)
- optional `errorReason` (string)

## Config schemas (recommended; logical)
- `service`: `id`, `name`, `resource`, optional `description`
- `x402_policy`: `id`, `serviceId`, `enabled`, `x402Version`, optional `message`
- `x402_accept`: `id`, `policyId`, `scheme`, `network`, `maxAmountRequired`, `payTo`, `asset`, optional `resource`, optional `description`, optional `maxTimeoutSeconds`

Deterministic resource rule (binding):
- If `x402_accept.resource` is present, it overrides.
- Otherwise use `service.resource`.

## Normalization (binding)
When ingesting, normalize into the micropayments `Transaction` shape (see `docs/spec/micropayments.md`):
- `sourceType = "x402"`
- `sourceRef` from `X-PAYMENT-RESPONSE.transaction` when present
- `amount` uses `maxAmountRequired` in v1
- proof stored as `proofHash` (default) or `rawProof` (optional)

## Proof storage (binding)
Bus MUST support storing either:
- `proofHash` (stable hash of canonical proof JSON), or
- `rawProof` (decoded JSON or base64)

Recommended default:
- store `proofHash` by default
