# Roadmap 0.0.6 — Schemas Module (Definition, Validation, Registration)

## Goal
Introduce schemas as a core-supported primitive while keeping schema parsing/validation behind interfaces so alternative implementations are possible.

## Progress
planned

## Non-goals
- Implement unit CRUD.
- Add schema migrations/versioning.

## User-visible outcomes
- A user can create/register a schema and validate it deterministically.

## Related previous versions
- `0.0.3` (manifest exists; workspace discovery rules)
- `0.0.4` (codec-backed document IO)
- `0.0.5` (locking/atomic writes for mutating operations)

## New or changed interfaces (exact names + boundaries)
Core-owned schema contracts (name and responsibility boundaries):
- `SchemaStore`: load schema documents (codec-agnostic).
- `SchemaRegistry`: map schema name → schema path (from manifest).
- `SchemaValidator`: validate schema structure and constraints.

Canonical rules live in:
- `docs/spec/schemas.md`

## Module responsibilities (what belongs in a module vs core)
- **Core**: owns the schema interfaces and document IO primitives.
- **Schemas module**: implements `bus schema init` and schema validation/registration behavior.

## Data model changes
- Manifest includes `units[]` schema references (see `docs/spec/schemas.md`).
- Schema documents use `kind: bus.schema` (see `docs/spec/schemas.md`).

## API/CLI changes
- `bus schema init <schemaName> [--path <file>] [--format yaml|toml|json] [--property <spec>]...`

## Configuration knobs introduced
None.

## Documentation goals
- Document schema format, types, and registration rules (single source: `docs/spec/schemas.md`).

## Implementation goals
- Implement schema init/register behavior and validation rules.
- Update manifest atomically when registering a schema.

## Automated testing goals
- Add schema validation tests (primary id rules, type rules, uniqueness rules).
- Add tests for atomic schema+manifest write behavior.

## Acceptance criteria
- Creating a schema writes a schema document and updates the manifest atomically.
- Invalid schema definitions fail with clear errors (e.g., missing/duplicate primary).
