# Roadmap 0.0.13 — Exports and Settlement Extension Points (Interfaces Only)

## Goal
Define the core extension points for turning transactions into external artifacts (reports/exports/settlement proposals) as module interfaces, without adding business logic to core.

## Progress
planned

## Non-goals
- Implement invoice generation or payment settlement flows.
- Define an accounting engine inside core.

## User-visible outcomes
- Users can list available exporters/proposers and run them deterministically (even if only a stub exists initially).

## Related previous versions
- `0.0.1` (capability registration exists)
- `0.0.10` (normalized transaction model exists for exporters/proposers to consume)

## New or changed interfaces (exact names + boundaries)
Core-owned extension point interfaces:
- `Exporter`
- `SettlementProposer`
- `ReportingView`

Canonical definitions live in:
- `docs/spec/exports-and-settlement.md`

## Module responsibilities (what belongs in a module vs core)
- **Core**: capability registration and request context enforcement.
- **Modules**: implement exporters/proposers/views and own their configs/output formats.

## Data model changes
None.

## API/CLI changes
Generic extension commands:
- `bus export list`
- `bus export run --exporter <ID> ...`
- `bus settlement propose --proposer <ID> ...`

## Configuration knobs introduced
Module-owned knobs (examples): exporter defaults, output format defaults, proposal period defaults.

## Documentation goals
- Document exporter/proposer/view interfaces and module boundaries (single source: `docs/spec/exports-and-settlement.md`).

## Implementation goals
- If the codebase exists: implement capability registration for listing and invoking exporters/proposers by id.
- If no implementation is planned yet: explicitly none (this version can be documentation-only), but then the user-visible “list/run” outcome must be treated as future.

## Automated testing goals
- If implementation is added in this version: add tests for listing selection by id and deterministic output handling.
- If no implementation is added: none (documentation-only version).

## Acceptance criteria
- Adding a new exporter/proposer requires only implementing the interface and registering a module capability.
- Core does not require code changes to add/select an exporter/proposer.
