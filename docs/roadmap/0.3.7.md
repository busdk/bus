# Roadmap 0.3.7 — Core: Database State Backend

## Goal
Add the database state backend as an additional built-in internal state implementation, selectable via manifest config.

## Progress
planned

## Non-goals
- Make DB backend required for the minimal path (filesystem `.bus` remains default).

## User-visible outcomes
- None (core-only).

## Related previous versions
- `0.1.4` (filesystem backend exists first)

## New or changed interfaces (exact names + boundaries)
Database backend requirements (canonical):
- `docs/spec/state-backend-database.md`

## Module responsibilities (what belongs in a module vs core)
- **Core**: selects DB backend when configured.
- **DB backend**: stores internal state transactionally and tenant-scoped.

## Data model changes
DB schema is backend-owned (versioned/migrated deterministically by the backend).

## API/CLI changes
None (core-only patch).

## Configuration knobs introduced
- Manifest: `state.backend: database`
- Manifest: `state.config`

## Documentation goals
- Ensure DB backend requirements are documented in `docs/spec/state-backend-database.md`.

## Implementation goals
- Implement DB backend with tenant scoping and transactional semantics.

## Automated testing goals
- Tests for tenant isolation and transactional atomicity in DB backend.

## Acceptance criteria
- DB backend works without requiring `.bus/`.
