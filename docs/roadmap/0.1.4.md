# Roadmap 0.1.4 — Core: Internal State Backend Interfaces + Filesystem Backend

## Goal
Introduce internal mutable state storage behind interfaces and implement the filesystem `.bus/state` backend as the first available backend.

## Progress
planned

## Non-goals
- Database backend.

## User-visible outcomes
- None (core-only).

## Related previous versions
- `0.1.1` (implementation selection pattern exists)

## New or changed interfaces (exact names + boundaries)
Internal state storage (canonical):
- `docs/spec/state-storage.md`
- `docs/spec/state-backend-dotbus.md`

## Module responsibilities (what belongs in a module vs core)
- **Core**: owns state interfaces and selects built-in backend (default filesystem).
- **Filesystem backend**: stores internal state under `.bus/state`.

## Data model changes
Creates `.bus/state` only when filesystem backend is selected.

## API/CLI changes
None (core-only patch).

## Configuration knobs introduced
- Manifest: `state.backend` (default: `filesystem`)
- Manifest: `state.config` (optional backend-specific config)

## Documentation goals
- Clarify `.bus` is only one backend option; DB backend comes later.

## Implementation goals
- Implement state backend interface + filesystem backend.

## Automated testing goals
- Unit tests that filesystem backend stores records deterministically.

## Acceptance criteria
- Internal state access is via interfaces; filesystem backend works as default.
