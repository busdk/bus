# Roadmap 0.0.5 — Workspace Locking and Atomic Writes

## Goal
Make every mutating operation atomic and safe under concurrency.

## Progress
planned

## Non-goals
- Implement any specific schema/unit commands.
- Provide automated repair tooling.

## User-visible outcomes
- Mutating commands never leave the workspace in an inconsistent state.
- Concurrent mutating commands are serialized by a tenant lock.

## Related previous versions
- `0.0.3` (tenant storage roots exist)
- `0.0.4` (document stores exist across formats)

## New or changed interfaces (exact names + boundaries)
Core-owned consistency primitives:
- `Locker`
- atomic write strategy
- multi-file commit-point rule

Canonical rules live in:
- `docs/spec/locking-and-atomic-writes.md`

## Module responsibilities (what belongs in a module vs core)
- **Core**: own locking and atomic write utilities; require core stores to use them.
- **Modules**: do not write files directly; use core stores and obey commit-point ordering.

## Data model changes
Inside `.bus/tenants/<tenantId>/`:
- `lock` — advisory lock file

## API/CLI changes
None (behavioral constraint on mutating commands).

## Configuration knobs introduced
None.

## Documentation goals
- Document locking and atomic write rules as binding constraints (single source: `docs/spec/locking-and-atomic-writes.md`).

## Implementation goals
- Implement tenant-scoped locking for mutating operations.
- Implement atomic write utilities and enforce commit-point ordering in stores.

## Automated testing goals
- Add tests that simulate failures mid-operation and verify no partial state is visible.
- Add tests for lock enforcement (concurrent mutators are serialized).

## Acceptance criteria
- Any mutating command acquires the tenant lock before writing.
- No file is partially written (temp+rename used everywhere).
- Multi-file operations update indexes/manifest last.
