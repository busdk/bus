# Roadmap 0.0.12 — x402 Module (Ingestion + Transaction Normalization)

## Goal
Specify how Bus ingests x402 proof headers and logs a normalized transaction record, with uniqueness rules to prevent double booking.

## Progress
planned

## Non-goals
- Verify on-chain state or integrate with external payment networks.
- Settlement and invoicing.

## User-visible outcomes
- `bus x402 ingest --service <SERVICE_ID> --x-payment <BASE64_JSON> [--x-payment-response <BASE64_JSON>]` records a transaction (or errors deterministically).

## Related previous versions
- `0.0.11` (x402 config + generation exists)
- `0.0.10` (normalized micropayments transaction model exists)

## New or changed interfaces (exact names + boundaries)
x402 ingestion capability:
- parse `X-PAYMENT` and optional `X-PAYMENT-RESPONSE` (base64 JSON)
- normalize into micropayments `Transaction`
- enforce uniqueness to prevent double booking
- store `proofHash` (default) or `rawProof` (optional)

Canonical definitions live in:
- `docs/spec/x402.md`
- `docs/spec/micropayments.md`

## Module responsibilities (what belongs in a module vs core)
- **Core**: unit creation primitives, locking, atomic writes, canonical serialization hook.
- **x402 module**: parsing + normalization + uniqueness + proof storage policy.

## Data model changes
None required in core. Transaction fields/uniqueness and proof storage rules are defined in spec files.

## API/CLI changes
- `bus x402 ingest --service <SERVICE_ID> --x-payment <BASE64_JSON> [--x-payment-response <BASE64_JSON>]`

## Configuration knobs introduced
Module-owned proof storage policy, e.g.:
- `x402.storeProof`: `hash|raw`

## Documentation goals
- Document x402 header parsing, normalization, and proof storage rules (single source: `docs/spec/x402.md`).

## Implementation goals
- Implement `bus x402 ingest` parsing base64 JSON headers.
- Implement normalization into the micropayments `Transaction` shape.
- Implement uniqueness enforcement to prevent double booking.
- Implement proof storage policy (hash default; raw optional) without changing core.

## Automated testing goals
- Add tests for header parsing errors (bad base64, bad JSON, missing required fields).
- Add tests for normalization field mapping and deterministic stored results.
- Add tests for uniqueness enforcement and replay/double-booking prevention.

## Acceptance criteria
- Ingestion is deterministic for the same inputs and workspace state.
- Uniqueness rules prevent double booking for supported identifier sets.
- Normalized transactions are stored using standard unit primitives.
