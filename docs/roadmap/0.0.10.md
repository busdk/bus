# Roadmap 0.0.10 — Micropayments Module (Normalized Transaction + Reporting)

## Goal
Introduce micropayments as a module layered on top of the append-only transaction pattern: capture sources normalize into a common transaction shape, and reporting aggregates over that shape.

## Progress
planned

## Non-goals
- Settlement/invoicing automation.
- Blockchain/network integration.

## User-visible outcomes
- `bus micropayments report ...` produces deterministic micropayment-oriented aggregates.

## Related previous versions
- `0.0.9` (append-only transaction pattern exists)

## New or changed interfaces (exact names + boundaries)
Micropayments module concepts standardized:
- normalized micropayments `Transaction` shape
- uniqueness requirements to prevent double booking
- reporting/aggregation capability

Canonical definitions live in:
- `docs/spec/micropayments.md`

## Module responsibilities (what belongs in a module vs core)
- **Core**: unit storage and deterministic encoding primitives.
- **Micropayments module**: define normalized transaction model and reporting.

## Data model changes
None required in core. The module defines required fields/constraints for the normalized transaction shape (see spec).

## API/CLI changes
- `bus micropayments report [--from <ts>] [--to <ts>] [--group-by <dim>]...`

## Configuration knobs introduced
None required.

## Documentation goals
- Document the normalized micropayments transaction shape and double-booking prevention requirements (single source: `docs/spec/micropayments.md`).

## Implementation goals
- Implement `bus micropayments report` over stored transaction records that match the normalized shape.
- Implement uniqueness enforcement required to prevent double booking (either via schema constraints, deterministic ids, or module-owned indexes under `.bus/`).

## Automated testing goals
- Add tests for deterministic reporting output and grouping semantics.
- Add tests that demonstrate double-booking prevention rules are enforced.

## Acceptance criteria
- Reporting groups/filters deterministically by at least: `fromUnitId`, `toUnitId`, `serviceId`, `asset`, `network`, and time window.
- Prevent double booking using the uniqueness rules defined in the spec.
