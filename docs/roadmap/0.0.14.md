# Roadmap 0.0.14 — Hosted Facilitator (HTTP) and OpenAPI

## Goal
Specify the hosted facilitator as a thin HTTP front-end over the same core/module operations as the CLI, with a published OpenAPI specification.

## Progress
planned

## Non-goals
- Implement authentication details (only define hook points).
- Add business logic to the HTTP layer.

## User-visible outcomes
- A future `busd` process can expose the same operations as the CLI over HTTP.
- The server publishes an OpenAPI document so clients can be generated.

## Related previous versions
- `0.0.2` (transport contract pattern established for CLI; HTTP should mirror)
- `0.0.11` + `0.0.12` (x402 endpoints reuse those module operations)
- `0.0.10` (micropayments reporting endpoint)

## New or changed interfaces (exact names + boundaries)
Transport-level concepts for HTTP routing and OpenAPI publication.

Canonical definitions live in:
- `docs/spec/hosted-facilitator-http.md`

## Module responsibilities (what belongs in a module vs core)
- **Core**: transport adapter mapping HTTP requests to `RequestContext` and enforcing tenant selection + hook points.
- **Modules**: register HTTP routes/endpoints as capabilities and provide request/response schemas for OpenAPI generation.

## Data model changes
None.

## API/CLI changes
HTTP surface and OpenAPI requirements are defined in the spec file (future-facing).

## Configuration knobs introduced
Transport-level knobs (examples): bind address, base URL, auth mode selection.

## Documentation goals
- Document HTTP surface and OpenAPI requirements as binding future contracts (single source: `docs/spec/hosted-facilitator-http.md`).

## Implementation goals
- None required in this repo right now (future transport).

## Automated testing goals
- None required in this repo right now (future transport).

## Acceptance criteria
- HTTP is a thin front-end: module capability logic is not duplicated.
- OpenAPI 3.x is published and fully describes endpoints and schemas.
