# Roadmap 0.0.8 — Relations (Typed References) and Scoped Uniqueness

## Goal
Define the v1 relations model as schema-driven typed references, plus scoped uniqueness conventions for safe modeling.

## Progress
planned

## Non-goals
- Introduce explicit relation entities.
- Add bidirectional relations or relation metadata beyond unit properties.

## User-visible outcomes
- Users can model provider/consumer relationships using `ref:<schema>` properties.
- Users can enforce “unique within org/tenant” constraints using `uniqueScope`.

## Related previous versions
- `0.0.7` (unit validation exists; ref/uniqueness enforcement is implementable)

## New or changed interfaces (exact names + boundaries)
No new core interfaces are required beyond schema/unit validation; this step standardizes semantics:
- typed references (`ref:<SchemaName>`)
- scoped uniqueness (`uniqueScope`)

Canonical rules live in:
- `docs/spec/relations.md`

## Module responsibilities (what belongs in a module vs core)
- **Core**: define typed reference semantics as part of the type system.
- **Units module**: enforce reference existence and scoped uniqueness during `add`.

## Data model changes
None (relations are stored in unit `data`).

## API/CLI changes
None.

## Configuration knobs introduced
None.

## Documentation goals
- Document the relations model and scoped uniqueness semantics (single source: `docs/spec/relations.md`).

## Implementation goals
- No additional implementation required if `0.0.7` already enforces typed references and `uniqueScope`.
- If `0.0.7` omitted any of these checks, implement them here (and cover them with tests in this same version).

## Automated testing goals
- If implementation changes are required in this version, add/extend tests for:
  - ref existence validation (tenant-scoped)
  - `uniqueScope` semantics
- If no implementation changes are required, no new tests are required (documented above).

## Acceptance criteria
- Creating a unit with a `ref:<schema>` property fails if the referenced unit id does not exist in the same tenant.
- `uniqueScope` constraints behave deterministically across YAML/TOML/JSON.
