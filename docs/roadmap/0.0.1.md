# Roadmap 0.0.1 — Bus Core Module Runtime (Registry, Lifecycle, Capabilities)

## Goal
Define Bus Core as a small orchestration layer that can load modules, wire dependencies, and expose module capabilities to transports without per-feature core changes.

## Progress
planned

## Non-goals
- Implement any concrete feature logic (schema/unit/x402/etc).
- Define persistence formats or workspace layouts.

## User-visible outcomes
- A minimal Bus process can start, load modules, and report which modules/capabilities are active.

## Related previous versions
- `0.0.0` (doc conventions, vocabulary, principles)

## New or changed interfaces (exact names + boundaries)
Core MUST define and own the module runtime contracts:
- `Module`, `Registrar`, `ModuleRegistry`, `ModuleRuntime`, `CapabilitySet`

Canonical definitions live in:
- `docs/spec/module-runtime.md`

## Module responsibilities (what belongs in a module vs core)
- **Core**: module registry/runtime lifecycle orchestration and dependency ordering.
- **Modules**: register capabilities and implementations; contain business logic.

## Data model changes
None.

## API/CLI changes
None required yet (introspection commands can be added in `0.0.2`).

## Configuration knobs introduced
`RuntimeConfig` (conceptual; manifest shape defined in `0.0.3`):
- enabled/disabled modules
- per-module config blob (validated by the module)

## Documentation goals
- Document the module runtime as the primary extension point (link to `docs/spec/module-runtime.md`).

## Implementation goals
- Implement module discovery/registration, dependency ordering, and a capability registry in core.

## Automated testing goals
- Add tests for deterministic module ordering and cycle detection.

## Acceptance criteria
- Runtime produces deterministic module load order.
- Cycles in dependencies are detected and reported.
- A module can register a capability without core branching changes.
