# Roadmap 0.0.9 — Ledger / Transactions as a Schema Pattern (Append‑Only)

## Goal
Define “transactions” as a schema-driven, append-only pattern using normal units, keeping the core small (no special billing engine).

## Progress
planned

## Non-goals
- Implement a billing engine, recurring generation, or invoicing.
- Add a special transaction subsystem distinct from units.

## User-visible outcomes
- Users can define a `transaction` schema and create ledger entries using normal unit commands.

## Related previous versions
- `0.0.6` (schema operations can declare create-only intent)
- `0.0.7` (units support create/list/show)

## New or changed interfaces (exact names + boundaries)
No new interfaces are required beyond schemas/units; this step standardizes the pattern and constraints.

Canonical rules live in:
- `docs/spec/ledger-transactions.md`

## Module responsibilities (what belongs in a module vs core)
- **Core**: remains storage/validation/orchestration only.
- **Modules**: own reporting/aggregation/export/settlement proposals on top of the ledger.

## Data model changes
None (a ledger is a schema + units).

## API/CLI changes
No new commands required beyond unit commands:
- `bus transaction add ...`
- `bus transaction list`
- `bus transaction show <id>`

## Configuration knobs introduced
None.

## Documentation goals
- Document the “transactions are units” pattern and the append-only rule (single source: `docs/spec/ledger-transactions.md`).

## Implementation goals
- No special subsystem is added. If the implementation supports an `operations` field on schemas, ensure it is preserved/validated.
- If any command attempts to “edit” a transaction, ensure the implementation does not provide that behavior (or rejects it) in this same version.

## Automated testing goals
- Add/extend tests verifying transaction schemas can be treated as create-only by convention (and by `operations` intent if present).
- Add tests that “correction” is represented as a new record (no updates performed by the tool).

## Acceptance criteria
- A ledger can be modeled entirely with schema+unit primitives.
- The append-only rule is explicit: corrections are new records.
