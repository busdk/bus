## Acceptance criteria

- The intended implementation order is the **SemVer sort order** (for this repo: `0.0.0`, `0.0.1`, …).

---

## SemVer roadmap rule (machine-parsable)
Each file under `docs/roadmap/{VERSION}.md` is the specification for Bus version `{VERSION}`.

Examples:
- `docs/roadmap/0.0.0.md` ⇒ Bus `0.0.0`
- `docs/roadmap/0.0.7.md` ⇒ Bus `0.0.7`

## Spec vs roadmap
- **Roadmap files (`docs/roadmap/`)**: step-by-step implementation plan. Each version is one small increment with acceptance criteria.
- **Spec files (`docs/spec/`)**: reusable definitions (one concept per file) referenced by roadmap steps.

## Progress field (binding)
Each roadmap file MUST include a `## Progress` section with one of:
- `planned`
- `in_progress`
- `done`

Definition:
- **planned**: specified but not implemented/verified yet
- **in_progress**: actively being implemented
- **done**: implementation + automated tests (if applicable) are complete for this version’s scope

## Vocabulary (single source)
Use the canonical terms defined in:
- `docs/spec/vocabulary.md`

## Non-negotiable architectural principles (single source)
These principles are binding for every roadmap step:
- `docs/spec/principles.md`
