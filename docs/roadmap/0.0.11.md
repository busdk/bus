# Roadmap 0.0.11 — x402 Module (Config Schemas + 402 Body Generation)

## Goal
Specify x402 as a capture mechanism module that stores config as units and generates the HTTP 402 “payment required” JSON body deterministically from that config.

## Progress
planned

## Non-goals
- Proof header ingestion (next version).
- Blockchain/network settlement.

## User-visible outcomes
- `bus x402 402 --service <SERVICE_ID>` prints the 402 JSON body.

## Related previous versions
- `0.0.7` (schemas/units exist to store config as units)
- `0.0.4` (deterministic JSON encoding rules)

## New or changed interfaces (exact names + boundaries)
x402 generation capability:
- config schemas (`service`, `x402_policy`, `x402_accept`)
- deterministic 402 body generation rules

Canonical definitions live in:
- `docs/spec/x402.md`

## Module responsibilities (what belongs in a module vs core)
- **Core**: deterministic JSON encoding primitives.
- **x402 module**: owns config schemas and generation rules.

## Data model changes
Introduces the recommended config schemas as normal schemas/units (see spec).

## API/CLI changes
- `bus x402 402 --service <SERVICE_ID>`

## Configuration knobs introduced
None.

## Documentation goals
- Document x402 config schemas and deterministic 402 generation rules (single source: `docs/spec/x402.md`).

## Implementation goals
- Implement `bus x402 402 --service <SERVICE_ID>` using unit-backed config.
- Implement deterministic resource selection and JSON output.

## Automated testing goals
- Add tests for deterministic 402 body generation given a fixed config state.
- Add tests for missing/disabled policy behavior (chosen behavior must be deterministic).

## Acceptance criteria
- Generating a 402 body is deterministic for a given config state.
- Missing/disabled policy causes a clear deterministic error (or empty accepts; pick one behavior and apply consistently).
