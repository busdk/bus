# Roadmap 0.0.7 — Units Module (Create/List/Show) and Tenant-Scoped Storage

## Goal
Implement the unit primitive as a module: create/list/show units that conform to schemas, stored as deterministic per-record files with merge-friendly indexes.

## Progress
planned

## Non-goals
- Unit update/delete.
- Advanced query/filtering.

## User-visible outcomes
- `bus <schema> add ...` creates a unit record.
- `bus <schema> list` lists unit ids.
- `bus <schema> show <id>` shows a unit record.

## Related previous versions
- `0.0.6` (schemas exist and are registered/validated)

## New or changed interfaces (exact names + boundaries)
Core-owned unit contracts:
- `UnitStore`: create/list/load units (tenant-scoped).
- `UnitValidator`: validate unit data against schema (types/refs/uniqueness).
- `PrimaryIDGenerator`: auto-generate UUID primary ids when omitted.

Canonical rules live in:
- `docs/spec/units.md`

## Module responsibilities (what belongs in a module vs core)
- **Core**: tenant-scoped storage roots, locking/atomic writes, codecs/deterministic encoding.
- **Units module**: implement create/list/show commands and validation.

## Data model changes
Tenant-scoped layout under `.bus/tenants/<tenantId>/`:
- `units/<schemaName>.ids`
- `units/<schemaName>/<primaryId>.<ext>`

See `docs/spec/units.md`.

## API/CLI changes
- `bus <SCHEMA_NAME> add key=value key2=value2 ...`
- `bus <SCHEMA_NAME> list`
- `bus <SCHEMA_NAME> show <primaryId>`
- compatibility: `bus unit <SCHEMA_NAME> ...`

## Configuration knobs introduced
None.

## Documentation goals
- Document unit document format, tenant-scoped storage layout, and create/list/show validation rules (single source: `docs/spec/units.md`).

## Implementation goals
- Implement unit create/list/show against tenant-scoped storage layout.
- Enforce type/ref/uniqueness validation during `add`.

## Automated testing goals
- Add tests for `key=value` parsing into unit data (unknown/duplicate keys error).
- Add tests for UUID autogeneration and index/record atomicity ordering.
- Add tests for ref validation and uniqueness enforcement (including `uniqueScope`).

## Acceptance criteria
- Unit creation is atomic: record file written first, index updated last.
- Validation enforces types, refs, and uniqueness (including `uniqueScope`).
- Behavior is tenant-scoped.
