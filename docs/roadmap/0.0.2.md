# Roadmap 0.0.2 — Transport Contracts (CLI First) and Command Routing

## Goal
Define a thin CLI transport contract so the CLI can route requests to module capabilities without embedding business logic in the transport or in core.

## Progress
planned

## Non-goals
- Define workspace file formats or persistence.
- Implement specific feature commands beyond minimal introspection.

## User-visible outcomes
- A minimal `bus` CLI can:
  - start the runtime
  - list loaded modules/capabilities
  - show which tenant context it is operating under

## Related previous versions
- `0.0.1` (module runtime + capability registration)

## New or changed interfaces (exact names + boundaries)
- `RequestContext`
- `Command`
- CLI command routing rules

Canonical definitions live in:
- `docs/spec/transports-cli.md`

## Module responsibilities (what belongs in a module vs core)
- **Core**: provide `RequestContext` creation rules and enforce tenant presence.
- **Modules**: register `Command` implementations; expose schema-routed unit command sets.

## Data model changes
None.

## API/CLI changes
Introduce minimal introspection commands (module-provided capabilities):
- `bus modules list`
- `bus capabilities list`

## Configuration knobs introduced
- CLI: `--tenant <TENANT_ID>` (defaulting rules defined in `0.0.3`)

## Documentation goals
- Document CLI routing + `key=value` parsing rules as binding behavior (see `docs/spec/transports-cli.md`).

## Implementation goals
- Implement a CLI router that resolves argv → registered `Command` capabilities.
- Implement `bus modules list` and `bus capabilities list` as module-provided commands.

## Automated testing goals
- Add tests for CLI routing order, collisions (built-in vs schema), and deterministic listing output.

## Acceptance criteria
- CLI routing follows the deterministic resolution + collision rules.
- Every command execution has a `RequestContext` containing a `TenantID`.
- Introspection output is deterministic (stable ordering).
