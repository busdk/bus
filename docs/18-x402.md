# x402 Micropayments (v1): Capture Mechanism Spec

x402-style micropayments are one **HTTP capture mechanism** for recording Bus **transactions** between business units/services.

Practically: an x402 “payment required / retry with proof” exchange is a way for a client to **purchase access** to a resource and for the service (or a hosted Bus facilitator) to **record the resulting transaction** in Bus.

High-level flow:
1. A service returns **HTTP 402** with a structured “payment requirement” JSON body.
2. A client selects an option, performs the payment action out-of-band (scheme-specific), and retries the request with payment proof in headers.
3. The service returns success with an `X-PAYMENT-RESPONSE` header.
4. Bus ingests these artifacts and logs a normalized **transaction** record for audit and reporting.

Bus v1 does **not**:
- integrate with blockchain networks
- confirm on-chain state
- automate invoicing or settlement

Bus v1 **does**:
- store x402 configuration for services
- generate the 402 JSON “payment requirement” body from config
- ingest `X-PAYMENT` and `X-PAYMENT-RESPONSE` headers
- log them as normalized transactions (see `17-micropayments.md`)

## Wire Formats (Versioned JSON; Spec-Only)

All wire bodies/headers use JSON objects with an explicit `x402Version` field (default `1`).

### A) 402 Payment Required Response Body

Response body is JSON:
- **x402Version** (number, default `1`)
- **accepts** (array of options)

Each option:
- **scheme** (string, e.g. `"exact"`)
- **network** (string)
- **maxAmountRequired** (string, base units)
- **resource** (string)
- **payTo** (string)
- **asset** (string)
- optional: **description** (string)
- optional: **maxTimeoutSeconds** (number)

Example (illustrative):

```json
{
  "x402Version": 1,
  "accepts": [
    {
      "scheme": "exact",
      "network": "testnet",
      "maxAmountRequired": "100",
      "resource": "GET /v1/widgets",
      "payTo": "merchant:example",
      "asset": "USD",
      "description": "Pay-per-request",
      "maxTimeoutSeconds": 30
    }
  ]
}
```

### B) Client Retry Proof Header: `X-PAYMENT`

Client includes an `X-PAYMENT` header containing **base64-encoded JSON**.

Decoded JSON fields:
- **x402Version**
- **scheme**
- **network**
- **payload** (object; scheme-specific)

Bus MUST:
- base64-decode the header
- parse JSON
- store the decoded object and either:
  - the raw base64 string, or
  - a stable hash (`proofHash`)
for audit purposes.

### C) Success Header: `X-PAYMENT-RESPONSE`

On success, server includes `X-PAYMENT-RESPONSE` containing **base64-encoded JSON**.

Decoded JSON fields:
- **success** (boolean)
- **transaction** (string; transaction id / external ref)
- **network** (string)
- **payer** (string)
- optional: **errorReason** (string)

Bus MUST ingest this header when present and associate it with the resulting normalized transaction record.

## Mapping to Bus Transactions (Normalization)

Bus Core MUST normalize x402 capture into a `transaction` record:
- `sourceType = "x402"`
- `network` from the decoded headers (and/or selected accept option)
- `asset` from the selected accept option
- `amount` from the selected accept option (`maxAmountRequired` in v1; future versions may support actual charged amount)
- `serviceId` from the invoked service config
- `sourceRef` from `X-PAYMENT-RESPONSE.transaction` when present
- `rawProof` or `proofHash` from the header artifacts (see below)

### Raw vs Hashed Proof Storage

To support audit and privacy, Bus MUST support storing either:
- **rawProof**: the raw decoded JSON (or base64 string), or
- **proofHash**: a stable hash of the canonical proof JSON

Recommended v1 behavior:
- Store a hash by default and allow raw storage via an explicit option (spec-only).

## Generation: “402 Body” From Config

Given:
- `service`
- `x402_policy`
- `x402_accept[]`

Bus Core MUST be able to generate the 402 response body:
- Filter to accept options under an enabled policy
- Apply the deterministic `resource` rule (see `17-micropayments.md`)
- Emit versioned JSON with stable ordering (see `12-formatting.md`)

## Ingestion: Logging an x402 Capture

Given:
- `serviceId` (or resolvable service identity)
- `X-PAYMENT` (required)
- `X-PAYMENT-RESPONSE` (recommended on success)

Bus Core MUST:
- validate and parse the headers
- enforce uniqueness constraints (prevent double booking) based on available identifiers:
  - at least \((network, transaction)\) when `transaction` exists
  - optionally \((sourceType, sourceRef)\) if `sourceRef` is used as the normalized key
- write a normalized `transaction` record

## Verification Hooks (Spec-Only)

x402 capture is compatible with simple verification primitives:
- hash the canonical proof object into `proofHash`
- include nonces/validity windows inside `payload` when a scheme supports it
- store idempotency keys derived from `(serviceId, resource, payer, nonce)` where available

These are verification hooks for integrity and replay protection; they are not blockchain settlement.
